# 网页足球经理游戏 MVP 产品规格

## 1. 产品目标

开发一款单机网页足球经理游戏 MVP。玩家管理一支球队，参与一个固定 20 队联赛，通过查看阵容、自动选择首发、模拟比赛、查看比分与积分榜，逐轮推进完整赛季。

第一版重点验证核心管理闭环是否成立，而不是追求完整足球经理复杂度。

## 2. 目标用户

- 喜欢足球经理、模拟经营、数值养成类游戏的玩家
- 希望在浏览器中快速开始、低学习成本体验联赛推进的玩家
- MVP 阶段主要面向单人离线体验

## 3. MVP 范围

### 3.1 核心闭环

1. 玩家进入游戏并拥有一支球队
2. 查看球队球员列表与能力
3. 点击自动选择首发阵容
4. 模拟当前轮比赛
5. 查看本场比分
6. 查看联赛积分榜变化
7. 进入下一轮并重复以上流程

### 3.2 首版明确不做

- 真实球员、真实俱乐部授权
- 转会市场
- 青训系统
- 复杂财政系统
- 多联赛、多级别升降级
- 3D/2D 实时比赛画面
- 联网对战
- 复杂战术板与逐分钟操作
- 球员伤病、停赛、合同、工资

## 4. 技术定位

- 应用类型：前端网页游戏
- 主要语言：TypeScript
- 数据存储：初期使用本地 JSON 数据与 localStorage
- 架构预留：后续可接入后端 API 与数据库
- 推荐形态：单页应用，按页面或视图切换游戏状态

## 6. 数据模型

### 6.1 Player

球员是阵容与比赛模拟的基础。

字段建议：

- `id`: 唯一 ID
- `name`: 虚构姓名
- `age`: 年龄
- `position`: 位置，取值为 `GK`、`DF`、`MF`、`FW`
- `teamId`: 所属球队 ID
- `overall`: 能力，1-200
- `isStarter`: 是否首发

### 6.2 Team

球队代表联赛参赛单位。

字段建议：

- `id`: 唯一 ID
- `name`: 虚构队名
- `shortName`: 简称
- `players`: 球员 ID 列表
- `isUserControlled`: 是否由玩家控制
- `primaryColor`: 球队主色，可用于 UI

### 6.3 Match

比赛代表某一轮中两队之间的结果。

字段建议：

- `id`: 唯一 ID
- `round`: 轮次
- `homeTeamId`: 主队 ID
- `awayTeamId`: 客队 ID
- `homeScore`: 主队进球数，未赛时为空
- `awayScore`: 客队进球数，未赛时为空
- `status`: `scheduled`、`played`

MVP 可只保存比分，不必生成完整比赛事件。

### 6.4 League

联赛保存赛季整体状态。

字段建议：

- `id`: 唯一 ID
- `name`: 联赛名
- `season`: 赛季名
- `teamIds`: 20 支球队 ID
- `matches`: 赛程 Match ID 列表
- `currentRound`: 当前轮次
- `totalRounds`: 总轮次

MVP 主客场双循环 38 轮。

### 6.5 Standing

积分榜由已赛比赛结果计算，也可以缓存。

字段建议：

- `teamId`: 球队 ID
- `played`: 已赛场次
- `wins`: 胜场
- `draws`: 平局
- `losses`: 负场
- `goalsFor`: 进球
- `goalsAgainst`: 失球
- `goalDifference`: 净胜球
- `points`: 积分

排序规则：

1. 积分高者在前
2. 净胜球高者在前
3. 进球数高者在前
4. 队名排序作为稳定兜底

## 7. 比赛模拟规则

### 7.1 自动首发规则

目标：一键选出 11 名首发。

推荐阵型：`4-4-2`

选择规则：

- 1 名 `GK`
- 4 名 `DF`
- 4 名 `MF`
- 2 名 `FW`
- 每个位置优先选择 `overall` 最高的球员
- 若某位置人数不足，自动生成能力为1的补位球员
- 自动选择后更新球员 `isStarter`

### 7.2 球队强度计算

基于首发球员的能力之和计算球队比赛强度。

### 7.3 进球数生成

MVP 不模拟实时过程，只生成最终比分。

推荐规则：

1. 分别计算主队、客队 `teamPower`
2. 主队获得轻微主场加成，例如 `+3`
3. 根据双方强度差计算期望进球
4. 加入随机波动
5. 将结果限制在合理比分范围

示例设计：

```text
baseGoals = 1.2
powerDiff = ownPower - opponentPower
expectedGoals = baseGoals + powerDiff / 35
expectedGoals = clamp(expectedGoals, 0.2, 3.2)
score = randomGoalByExpectedValue(expectedGoals)
```

随机进球分布建议：

- 低期望值时更容易得到 0-1 球
- 中期望值时常见 1-2 球
- 高期望值时有机会得到 3-5 球
- 单队单场进球 MVP 阶段可限制在 0-5

### 7.4 电脑球队比赛

当前轮除玩家比赛外，其他球队比赛也应自动模拟。

规则：

- 每场未赛比赛调用同一模拟器
- 电脑球队可自动按能力选出首发
- 模拟完成后统一更新积分榜

## 8. 页面结构

### 8.1 DashboardPage 首页 / 轮次概览

展示：

- 当前联赛名与赛季
- 当前轮次
- 玩家球队基本信息
- 下一场对手
- 主要操作入口：阵容、模拟比赛、积分榜

主要操作：

- 进入球队阵容
- 进入当前比赛
- 查看积分榜

### 8.2 SquadPage 球队阵容

展示：

- 球员列表
- 姓名、年龄、位置、能力
- 当前首发标识

主要操作：

- 自动选择首发
- 查看首发阵容结果

MVP 不需要手动拖拽调整阵容。

### 8.3 MatchPage 比赛页

展示：

- 当前轮玩家比赛信息
- 主队与客队名称
- 比赛状态
- 已赛后的最终比分

主要操作：

- 模拟本轮比赛
- 比赛结束后进入积分榜或下一轮

### 8.4 StandingsPage 积分榜

展示：

- 排名
- 球队
- 场次
- 胜 / 平 / 负
- 进球 / 失球 / 净胜球
- 积分

主要操作：

- 返回首页
- 当前轮全部比赛完成后进入下一轮

### 8.5 SeasonEndPage 赛季结束页

当 `currentRound > totalRounds` 时展示。

展示：

- 最终排名
- 玩家球队排名
- 冠军球队
- 赛季简短总结

MVP 可只提供重新开始入口。

## 9. 游戏状态流转

```text
初始化游戏
  -> 生成 20 队与球员
  -> 生成赛程
  -> 当前轮 = 1
  -> 首页

首页
  -> 查看阵容
  -> 自动选择首发
  -> 当前比赛
  -> 模拟本轮
  -> 查看比分
  -> 更新积分榜
  -> 查看积分榜
  -> 进入下一轮

重复直到赛季结束
  -> 展示赛季结束页
```

状态保存：

- 每次完成比赛模拟后保存到 localStorage
- 进入下一轮后保存到 localStorage
- 提供“重新开始游戏”清空存档

## 10. 开发里程碑

### 里程碑 1：项目初始化与静态数据

内容：

- 建立 TypeScript 前端项目
- 定义核心数据模型
- 准备 20 支虚构球队
- 每队生成基础球员数据

验收标准：

- 项目可启动
- 能在页面中读取并展示球队与球员数据
- 至少存在 20 支球队
- 玩家球队至少有 11 名球员

### 里程碑 2：球队阵容与自动首发

内容：

- 实现球员列表页面
- 实现按能力自动选择首发
- 实现首发标识展示

验收标准：

- 玩家可以查看完整阵容
- 点击自动选择后，系统选出 11 名首发
- 首发位置结构符合 `4-4-2`
- 位置不足时仍能补齐 11 人

### 里程碑 3：赛程与轮次系统

内容：

- 生成 20 队单循环赛程
- 展示当前轮玩家比赛
- 支持当前轮状态判断

验收标准：

- 联赛包含 38 轮
- 每轮包含 10 场比赛
- 每支球队每轮最多参加 1 场比赛
- 首页能显示当前轮与下一场对手

### 里程碑 4：比赛模拟

内容：

- 实现比赛模拟器
- 玩家比赛与电脑比赛使用同一规则
- 生成比分并保存比赛结果

验收标准：

- 点击模拟后当前轮所有比赛完成
- 每场比赛都有合理比分
- 已完成比赛不会被重复模拟覆盖
- 玩家能看到本场比分

### 里程碑 5：积分榜

内容：

- 根据比赛结果计算积分榜
- 实现积分榜页面
- 支持按积分、净胜球、进球排序

验收标准：

- 胜平负积分正确：胜 3 分，平 1 分，负 0 分
- 进球、失球、净胜球计算正确
- 排名顺序符合规则
- 模拟一轮后积分榜发生正确变化

### 里程碑 6：本地存档与赛季推进

内容：

- 保存联赛状态到 localStorage
- 支持进入下一轮
- 支持刷新页面后恢复进度
- 支持赛季结束页

验收标准：

- 模拟比赛后刷新页面不会丢失结果
- 当前轮完成后才能进入下一轮
- 完成最后一轮后进入赛季结束页
- 可以重新开始新赛季

### 里程碑 7：MVP 打磨

内容：

- 补齐基础 UI 状态
- 增加空状态、禁用状态、错误状态
- 优化移动端与桌面端布局
- 做基础测试与手动验收

验收标准：

- 主要流程可以从新游戏走到赛季结束
- 页面无明显布局错乱
- 关键按钮在非法状态下不可点击
- 无控制台致命错误

## 11. MVP 成功标准

MVP 完成时，玩家应能：

- 打开网页开始一局新赛季
- 管理一支虚构球队
- 查看球员能力
- 一键生成首发
- 模拟每一轮比赛
- 查看比分和积分榜
- 推进并完成整个 20 队联赛赛季

技术上应满足：

- 核心规则集中在可测试的 TypeScript 逻辑模块中
- UI 与游戏规则分离
- 数据结构便于后续迁移到后端数据库
- 本地存档稳定可用

## 13. 建议第一版实施顺序

1. 先完成静态数据、模型和页面骨架
2. 再完成自动首发与比赛模拟
3. 然后接入积分榜与轮次推进
4. 最后加入 localStorage 存档和赛季结束

原因：先跑通“阵容 -> 比赛 -> 积分榜 -> 下一轮”的闭环，再逐步增强表现层与长期玩法。

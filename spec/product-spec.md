# 网页足球经理游戏 MVP 产品规格

## 1. 产品目标

开发一款单机网页足球经理游戏 MVP。玩家管理一支球队，参与一个包含 2 级联赛的固定联赛体系。每级联赛 20 支球队，通过查看阵容、自动选择首发、模拟比赛、查看比分与积分榜，逐轮推进完整赛季，并在赛季结束后根据排名完成升降级。

第一版重点验证核心管理闭环是否成立，而不是追求完整足球经理复杂度。

## 2. 目标用户

- 喜欢足球经理、模拟经营、数值养成类游戏的玩家
- 希望在浏览器中快速开始、低学习成本体验联赛推进的玩家
- MVP 阶段主要面向单人离线体验

## 3. MVP 范围

### 3.1 核心闭环

1. 新游戏开始时，玩家从最低级联赛中随机获得一支球队
2. 查看球队球员列表与能力
3. 点击自动选择首发阵容
4. 模拟当前轮比赛
5. 查看本场比分
6. 查看联赛积分榜变化
7. 进入下一轮并重复以上流程
8. 赛季结束后执行升降级并进入新赛季

### 3.2 首版明确不做

- 真实球员、真实俱乐部授权
- 青训系统
- 3D/2D 实时比赛画面
- 联网对战
- 复杂战术板与逐分钟操作
- 球员伤病、停赛
- 复杂财政细项，例如赞助、税务、贷款、球场维护
- 复杂合同条款，例如解约金、奖金、续约谈判

### 3.3 联赛体系范围

MVP 纳入两级联赛，但保持规则简单：

- 固定 2 个级别：一级联赛与二级联赛
- 每级联赛固定 20 支球队，共 40 支球队
- 每级联赛内部进行主客场双循环，共 38 轮
- 一级联赛赛季末后 3 名降入二级联赛
- 二级联赛赛季末前 3 名升入一级联赛
- 二级联赛没有更低级别，因此不再降级
- 新游戏时玩家球队从二级联赛 20 支球队中随机挑选

## 4. 技术定位

- 应用类型：前端网页游戏
- 主要语言：TypeScript
- 数据存储：初期使用本地 JSON 数据与 localStorage
- 架构预留：后续可接入后端 API 与数据库
- 推荐形态：单页应用，按页面或视图切换游戏状态

## 6. 数据模型

### 6.1 Player

球员是阵容与比赛模拟的基础。

字段建议：

- `id`: 唯一 ID
- `name`: 虚构姓名
- `age`: 年龄
- `position`: 位置，取值为 `GK`、`DF`、`MF`、`FW`
- `teamId`: 所属球队 ID
- `overall`: 能力，1-200
- `potential`: 潜力值，1-200，表示球员能力上限
- `marketValue`: 球员身价，整数金额
- `weeklyWage`: 球员周薪，每场比赛后发放
- `contractYears`: 合同剩余年限，1-5 年，且不能超过距离 36 岁退役的剩余赛季数
- `isListed`: 是否进入转会市场
- `isStarter`: 是否首发

退役规则：

- 常规球员进入新赛季时若年龄达到 36 岁，则退役并从球队移除
- 补位球员不参与退役结算

### 6.2 Team

球队代表联赛参赛单位。

字段建议：

- `id`: 唯一 ID
- `name`: 虚构队名
- `shortName`: 简称
- `leagueId`: 当前所属联赛 ID
- `players`: 球员 ID 列表
- `balance`: 现金余额
- `stadiumCapacity`: 球场容量
- `ticketPrice`: 单张门票价格
- `fanBase`: 球迷基础，影响上座率
- `isUserControlled`: 是否由玩家控制
- `primaryColor`: 球队主色，可用于 UI

### 6.3 Match

比赛代表某一轮中两队之间的结果。

字段建议：

- `id`: 唯一 ID
- `leagueId`: 所属联赛 ID
- `round`: 轮次
- `homeTeamId`: 主队 ID
- `awayTeamId`: 客队 ID
- `homeScore`: 主队进球数，未赛时为空
- `awayScore`: 客队进球数，未赛时为空
- `status`: `scheduled`、`played`

MVP 可只保存比分，不必生成完整比赛事件。

### 6.4 League

联赛保存赛季整体状态。

字段建议：

- `id`: 唯一 ID
- `name`: 联赛名
- `level`: 联赛级别，数值越小级别越高，例如 1 为一级联赛，2 为二级联赛
- `season`: 赛季名
- `teamIds`: 20 支球队 ID
- `matches`: 赛程 Match ID 列表
- `currentRound`: 当前轮次
- `totalRounds`: 总轮次

MVP 主客场双循环 38 轮。

### 6.5 LeagueSystem

联赛体系保存多个联赛之间的层级关系与赛季规则。

字段建议：

- `id`: 唯一 ID
- `season`: 当前赛季名
- `leagueIds`: 当前体系中的联赛 ID 列表
- `promotionSlots`: 升级名额，MVP 固定为 3
- `relegationSlots`: 降级名额，MVP 固定为 3
- `lowestLevel`: 最低级别，MVP 固定为 2

### 6.6 Standing

积分榜由已赛比赛结果计算，也可以缓存。

字段建议：

- `teamId`: 球队 ID
- `played`: 已赛场次
- `wins`: 胜场
- `draws`: 平局
- `losses`: 负场
- `goalsFor`: 进球
- `goalsAgainst`: 失球
- `goalDifference`: 净胜球
- `points`: 积分

排序规则：

1. 积分高者在前
2. 净胜球高者在前
3. 进球数高者在前
4. 队名排序作为稳定兜底

### 6.7 GameState

游戏状态需要同时保存两级联赛的赛程与玩家球队。

字段建议：

- `leagueSystem`: 当前联赛体系状态
- `leagues`: 2 个联赛对象
- `teams`: 40 支球队
- `players`: 所有球队球员
- `matches`: 两级联赛的全部比赛
- `userTeamId`: 玩家控制球队 ID
- `transferMarket`: 当前转会市场状态
- `financeLogs`: 财政流水，可记录门票收入、工资支出、转会支出与收入
- `seasonHomeIncomeByLeague`: 本赛季每个联赛锁定的单场主场收入
- `lastRetiredPlayerIds`: 最近一次新赛季结算退役球员 ID 列表，可用于展示
- `lastYouthPlayerIds`: 最近一次新赛季补充年轻球员 ID 列表，可用于展示

新游戏初始化规则：

1. 生成 2 个联赛，每级 20 支球队
2. 给每支球队生成 25 名基础常规球员
3. 从最低级联赛的 20 支球队中随机选择 1 支作为玩家球队
4. 将该球队标记为 `isUserControlled = true`
5. 分别为每级联赛生成 38 轮赛程
6. 为每名球员生成身价、周薪与受退役年龄约束的合同年限
7. 为每支球队生成初始现金余额、球场容量、票价与球迷基础
8. 按赛季初工资总额计算并锁定 `seasonHomeIncomeByLeague`

新赛季补员规则：

- 赛季结算、退役和升降级完成后，检查每支球队常规球员数量
- 若球队少于 25 名常规球员，自动生成 16 岁年轻球员补足空位
- 自动补员球员应拥有完整字段：`overall`、`potential`、`marketValue`、`weeklyWage`、`contractYears`、`isListed`
- 自动补员优先补齐位置结构，避免球队缺少门将或明显无法自动首发

### 6.8 TransferMarket

转会市场保存当前可购买球员列表。

字段建议：

- `listedPlayerIds`: 挂牌球员 ID 列表
- `lastRefreshRound`: 最近一次刷新轮次
- `lastRefreshSeason`: 最近一次刷新赛季

MVP 允许玩家从市场购买球员，也允许玩家从阵容中直接卖出球员。不需要实现电脑队主动报价、竞价、谈判失败、球员意愿等复杂逻辑。

### 6.9 FinanceLog

财政流水用于解释球队余额变化。

字段建议：

- `id`: 唯一 ID
- `season`: 赛季
- `round`: 轮次，可为空
- `teamId`: 球队 ID
- `type`: `ticketIncome`、`wageExpense`、`transferFee`
- `amount`: 金额，收入为正，支出为负
- `description`: 简短说明

## 7. 比赛模拟规则

### 7.1 自动首发规则

目标：一键选出 11 名首发。

推荐阵型：`4-4-2`

选择规则：

- 1 名 `GK`
- 4 名 `DF`
- 4 名 `MF`
- 2 名 `FW`
- 每个位置优先选择 `overall` 最高的球员
- 若某位置人数不足，自动生成能力为1的补位球员
- 自动选择后更新球员 `isStarter`

### 7.2 球队强度计算

基于首发球员的能力之和计算球队比赛强度。

### 7.3 进球数生成

MVP 不模拟实时过程，只生成最终比分。

推荐规则：

1. 分别计算主队、客队 `teamPower`
2. 主队获得轻微主场加成，例如 `+3`
3. 根据双方强度差计算期望进球
4. 加入随机波动
5. 将结果限制在合理比分范围

示例设计：

```text
baseGoals = 1.2
powerDiff = ownPower - opponentPower
expectedGoals = baseGoals + powerDiff / 35
expectedGoals = clamp(expectedGoals, 0.2, 3.2)
score = randomGoalByExpectedValue(expectedGoals)
```

随机进球分布建议：

- 低期望值时更容易得到 0-1 球
- 中期望值时常见 1-2 球
- 高期望值时有机会得到 3-5 球
- 单队单场进球 MVP 阶段可限制在 0-5

### 7.4 电脑球队比赛

当前轮除玩家比赛外，其他球队比赛也应自动模拟。MVP 中点击“模拟本轮”时，应推进所有联赛的同一轮次。

规则：

- 每个联赛当前轮的每场未赛比赛调用同一模拟器
- 电脑球队可自动按能力选出首发
- 模拟完成后分别更新各联赛积分榜

### 7.5 升降级与新赛季

赛季结束条件：

- 所有联赛的 `currentRound > totalRounds`
- 或者两级联赛的全部比赛状态均为 `played`

升降级规则：

1. 根据一级联赛最终积分榜，取后 3 名作为降级球队
2. 根据二级联赛最终积分榜，取前 3 名作为升级球队
3. 将升级球队的 `leagueId` 改为一级联赛 ID
4. 将降级球队的 `leagueId` 改为二级联赛 ID
5. 生成新赛季名称，例如 `2026/27`
6. 清空旧赛程并按新联赛归属重新生成两级联赛赛程
7. 将两级联赛 `currentRound` 重置为 1

新赛季结算顺序：

1. 所有常规球员年龄 `age + 1`
2. 所有常规球员合同年限 `contractYears - 1`，最低为 0
3. 年龄达到 36 岁的常规球员退役，从 `players` 和所属球队 `players` 中移除
4. 执行升降级并更新球队 `leagueId`
5. 对每支球队检查常规球员数量，少于 25 人则自动补充 16 岁年轻球员
6. 为补充的年轻球员生成完整财务与合同字段
7. 按新赛季球队和球员重新计算赛程、转会市场与 `seasonHomeIncomeByLeague`

边界规则：

- 一级联赛没有升级去向，只显示争冠与降级状态
- 二级联赛没有更低级别，后 3 名仅作为低排名展示，不执行降级
- 玩家球队如果升入一级联赛，下赛季继续由玩家控制
- 玩家球队如果从一级联赛降入二级联赛，下赛季继续由玩家控制
- MVP 不需要处理球队新增、解散或跨体系补位
- 退役可能发生在玩家球队；若玩家球队出现空位，也按同一规则自动补充 16 岁年轻球员

### 7.6 球员成长与衰退

球员能力变化在每场比赛模拟完成后结算；年龄在赛季结束并进入新赛季时统一结算。MVP 不需要训练系统，但每场比赛后都应让所有球员都有机会产生轻微能力变化。

新增概念：

- `potential` 是球员能力硬上限，范围与 `overall` 一致，为 1-200
- `potential` 创建后不会变动
- `overall` 在任何情况下都不能超过 `potential`
- `age` 只在赛季结束时统一增加 1
- 年轻球员更可能接近潜力值
- 巅峰期球员能力变化较小
- 高龄球员更可能衰退
- 36 岁球员在进入新赛季时退役
- 球员成长或衰退后，`overall` 仍需限制在 1-200
- 成长不应让 `overall` 超过 `potential`
- 衰退可以让 `overall` 低于历史能力，但不会改变 `potential`

年龄段规则：

| 年龄 | 阶段 | 单场赛后能力变化倾向 |
| --- | --- | --- |
| 16-20 | 新秀期 | 小概率成长，赛季累计成长明显 |
| 21-24 | 成长期 | 中等概率成长，接近潜力时放缓 |
| 25-29 | 巅峰期 | 大多数比赛保持不变，小概率波动 |
| 30-32 | 后巅峰期 | 小概率下降 |
| 33+ | 衰退期 | 较高概率下降 |

单场赛后结算建议公式：

```text
growthRoom = potential - overall

if age <= 20:
  delta = weightedRandom({ 0: 70%, 1: 25%, 2: 5% }) + floor(growthRoom / 120)
else if age <= 24:
  delta = weightedRandom({ 0: 78%, 1: 20%, 2: 2% }) + floor(growthRoom / 160)
else if age <= 29:
  delta = weightedRandom({ -1: 5%, 0: 88%, 1: 7% })
else if age <= 32:
  delta = weightedRandom({ -1: 18%, 0: 78%, 1: 4% })
else:
  delta = weightedRandom({ -2: 8%, -1: 32%, 0: 60% })

newOverall = clamp(overall + delta, 1, potential)
```

边界规则：

- 每场比赛后结算所有球员的能力变化，不限是否进入首发或出场
- 同一球员同一轮最多结算 1 次，避免跨联赛或重复模拟导致重复成长
- 已完成比赛不会被重复模拟，因此不会重复触发成长结算
- 每场比赛后的能力变化不改变球员年龄
- 赛季结束并开启新赛季时，所有常规球员 `age + 1`
- 年龄增长后达到 36 岁的常规球员退役
- 若历史数据或导入数据出现 `overall > potential`，初始化或读档修正时应将 `overall` 压回 `potential`
- MVP 不引入训练、出场时间、伤病、隐藏属性等影响因子

展示规则：

- 阵容页展示 `overall` 与 `potential`
- 比赛结束页展示玩家球队本场主要能力变化摘要
- 摘要至少列出本场玩家球队能力发生变化最大的若干球员；若无人变化，展示“本场无明显能力变化”
- 非玩家球队也执行成长结算，但 MVP 可不展示详情

### 7.7 退役与年轻球员补充

退役规则：

- 新赛季结算时，先执行年龄增长，再判断退役
- 常规球员年龄达到 36 岁时退役
- 退役球员从全局 `players` 移除
- 退役球员 ID 从所属球队 `players` 移除
- 退役球员若在首发中，也随移除自然取消首发
- 退役不产生转会费、工资补偿或财政流水

年轻球员补充规则：

- 新赛季开始时，每支球队常规球员数量应补足到 25 人
- 只有球队常规球员少于 25 人时才自动补员
- 自动补员球员年龄固定为 16 岁
- 自动补员球员属于常规球员，不是补位球员
- 自动补员球员应加入球队 `players` 列表，并写入全局 `players`
- 自动补员球员 `isStarter = false`
- 自动补员球员 `isListed` 可按普通新球员概率生成，或 MVP 固定为 `false`
- 自动补员后，球队仍可通过“最强阵容”重新选择首发

补员位置规则：

- 优先检查球队现有常规球员的位置结构
- 若门将少于 3 人，优先补 `GK`
- 然后按推荐阵容规模补齐：`DF 8`、`MF 9`、`FW 5`
- 若所有位置都达到推荐数量但仍有空位，则补当前人数最少的位置

年轻球员能力建议：

```text
age = 16
overall = random(20, 55)
potential = clamp(overall + random(25, 70), overall, 200)
contractYears = randomInt(2, 5)
contractYears = min(contractYears, 36 - age)
marketValue = calculateMarketValue(age, overall, potential)
weeklyWage = calculateWeeklyWage(overall)
```

边界规则：

- 自动补员不能让球队超过 25 名常规球员
- 自动补员后每支球队至少应有 2 名门将
- 若退役导致玩家球队低于 11 人，也必须先补足到 25 人再进入新赛季
- 补员发生在赛季初，补员球员不会参与上一赛季统计
- 补员完成后需重新计算新赛季转会市场与主场收入基准

### 7.8 转会与财政

MVP 加入轻量转会市场与基础财政。目标是让玩家能用球队资金购买球员，并在每场比赛后看到门票收入与工资支出对余额的影响。

#### 7.8.1 球员身价

球员身价由能力、潜力、年龄共同决定，生成后可随能力变化重新计算或在每轮后刷新。

建议公式：

```text
baseValue = overall * overall * 100
potentialBonus = max(0, potential - overall) * 5000
ageModifier =
  age <= 21 ? 1.25 :
  age <= 25 ? 1.15 :
  age <= 29 ? 1.00 :
  age <= 32 ? 0.75 :
  0.45

marketValue = round((baseValue + potentialBonus) * ageModifier)
```

边界规则：

- `marketValue` 最低为 1000
- `marketValue` 不能为负数
- 球员能力变化后，身价可在比赛结算后同步刷新
- `potential` 不因身价变化而变化

#### 7.8.2 球员周薪

周薪是每场比赛后发放的工资支出。

建议公式：

```text
abilityBand = ceil(clamp(overall, 1, 200) / 10)
weeklyWage = abilityBand * 1000
```

发放规则：

- 每场比赛模拟完成后，每支球队支付本队全部常规球员周薪
- 能力 1-10 的最低档球员周薪为 1000
- 能力段越高，周薪越高
- 周薪由当前能力档决定，不直接由身价决定
- 工资支出从球队 `balance` 中扣除
- 若球队余额为负，MVP 允许继续比赛，但余额展示为负数
- 工资支出写入 `financeLogs`

#### 7.8.3 合同年限

合同年限表示球员剩余合同长度。

规则：

- 新生成球员的 `contractYears` 基础范围为 1-5 年
- 合同年限必须受 36 岁退役约束，不能超过球员距离退役的剩余赛季数
- 计算公式：`maxContractYears = clamp(36 - age, 0, 5)`
- 生成公式：`contractYears = randomInt(1, maxContractYears)`
- 例如 35 岁球员最多只能拥有 1 年合同，34 岁球员最多 2 年合同
- 若 `maxContractYears <= 0`，该球员不应作为常规球员生成或进入新赛季，应执行退役逻辑
- 每个赛季结束并进入新赛季时，常规球员 `contractYears - 1`
- MVP 中合同到期不会自动离队，最低显示为 0
- 第一版不做续约谈判、合同奖金、解约金

边界规则：

- 初始化、年轻球员补充、读档修正、球员年龄增长后，都应保证 `contractYears <= max(0, 36 - age)`
- 若历史数据中出现合同跨过 36 岁退役年龄，应在读档或新赛季结算时压缩到允许上限
- 16 岁补充球员仍可使用 2-5 年合同，因为距离退役年龄足够远

#### 7.8.4 主场收入

主场收入在赛季初计算并锁定，之后在主场比赛结束后发放。MVP 先用赛季初联赛工资水平反推收入，保证收入能覆盖支出。

建议公式：

```text
teamWeeklyWage = sum(team.players.weeklyWage)
leagueMaxWeeklyWage = max(league.teams.teamWeeklyWage)
homeIncome = round(leagueMaxWeeklyWage * 2.5)
```

简化规则：

- 每个赛季开始时，每个联赛独立计算一次 `leagueMaxWeeklyWage`
- 一级联赛最高工资队决定一级联赛单场主场收入
- 二级联赛最高工资队决定二级联赛单场主场收入
- 计算结果保存为 `seasonHomeIncomeByLeague`
- 同一赛季内主场收入不随成长、衰退、转会或工资变化重新计算
- 每场比赛只给主队发放其所在联赛的主场收入
- 主场收入加入球队 `balance`
- 主场收入写入 `financeLogs`，类型仍使用 `ticketIncome`
- `stadiumCapacity`、`ticketPrice`、`fanBase` 暂时作为后续扩展参数保留

#### 7.8.5 转会市场

市场规则：

- 转会市场展示若干可购买球员
- 市场球员来自非玩家球队中 `isListed = true` 的球员
- 市场每轮刷新一次，或在进入新赛季时刷新
- 玩家购买球员时支付 `marketValue`
- 买方球队 `balance -= marketValue`
- 卖方球队 `balance += marketValue`
- 球员 `teamId` 改为玩家球队 ID
- 玩家球队 `players` 加入该球员 ID
- 原球队 `players` 移除该球员 ID
- 购买后该球员 `isListed = false`
- 转会支出和收入写入 `financeLogs`

购买限制：

- 玩家余额不足时不可购买
- 玩家球队已有 25 名常规球员时不可购买
- 电脑球队至少保留 11 名常规球员，避免被买空

卖出规则：

- 玩家可以从阵容页或转会页选择本队常规球员并执行卖出
- 卖出价格默认使用该球员当前 `marketValue`
- 玩家球队 `balance += marketValue`
- 球员离开玩家球队，MVP 可直接转移到系统买家或从玩家阵容中移除
- 玩家球队 `players` 移除该球员 ID
- 被卖出的球员 `teamId` 改为系统买家 ID 或空闲球员池 ID
- 被卖出的球员 `isStarter = false`
- 被卖出的球员 `isListed = false`
- 转会收入写入 `financeLogs`

卖出限制：
- 玩家球队卖出后常规球员少于 11 人时不可卖出
- 为避免阵容结构失效，卖出后若某位置低于自动首发最低需求，可允许卖出但下一次自动首发会生成补位球员；MVP 不要求强制阻止
- 正在首发的球员允许卖出，卖出成功后自动取消其首发状态

展示规则：

- 阵容页展示球员身价、周薪、合同年限
- 首页展示玩家球队余额
- 比赛结束页展示本轮门票收入、工资支出和余额变化摘要
- 转会市场页展示可买球员、所属球队、位置、年龄、能力、潜力、身价、周薪、合同年限
- 阵容页或转会页展示玩家球员的卖出按钮、卖出金额和不可卖原因

## 9. 游戏状态流转

```text
初始化游戏
  -> 生成 2 级联赛
  -> 每级生成 20 队与球员
  -> 从最低级联赛随机选择玩家球队
  -> 为每级联赛生成赛程
  -> 两级联赛当前轮 = 1
  -> 首页

首页
  -> 查看阵容
  -> 自动选择首发
  -> 当前比赛
  -> 模拟所有联赛本轮
  -> 查看比分
  -> 结算所有球员能力成长与衰退
  -> 结算主队门票收入
  -> 结算所有球队球员周薪
  -> 更新积分榜
  -> 查看积分榜
  -> 可进入转会市场购买球员
  -> 可从阵容或转会页卖出球员
  -> 进入下一轮

重复直到赛季结束
  -> 展示赛季结束页
  -> 结算球员年龄
  -> 结算合同年限
  -> 36 岁常规球员退役
  -> 执行升降级
  -> 球队空位自动补充 16 岁年轻球员至 25 人
  -> 生成新赛季赛程
  -> 进入新赛季首页
```

状态保存：

- 每次完成比赛模拟后保存到 localStorage
- 每场比赛后的球员能力成长与衰退随比赛结果一起保存
- 每场比赛后的门票收入、工资支出和球队余额变化保存到 localStorage
- 每次买入或卖出转会完成后保存到 localStorage
- 进入下一轮后保存到 localStorage
- 赛季结束年龄结算后保存到 localStorage
- 赛季结束合同年限结算后保存到 localStorage
- 赛季结束退役与年轻球员补员结果保存到 localStorage
- 执行升降级并生成新赛季后保存到 localStorage
- 提供“重新开始游戏”清空存档

- 进入新赛季时年龄达到 36 岁的常规球员会退役并从球队移除
- 退役后球队若少于 25 名常规球员，会自动补充 16 岁年轻球员至 25 人
- 自动补充的年轻球员拥有完整能力、潜力、身价、周薪和合同字段
- 自动补充后每支球队至少有 2 名门将
- 赛季中单场比赛后的能力结算不会提前改变球员年龄
- 一级联赛后 3 名下赛季进入二级联赛
- 二级联赛前 3 名下赛季进入一级联赛
- 两级联赛新赛季仍各有 20 支球队
- 两级联赛新赛季每支球队仍有 25 名常规球员
- 玩家球队无论升降级都继续由玩家控制
- 新赛季从第 1 轮开始，旧赛季比分不会污染新赛季赛程

### 里程碑 9：MVP 打磨

内容：

- 补齐基础 UI 状态
- 增加空状态、禁用状态、错误状态
- 优化移动端与桌面端布局
- 做基础测试与手动验收

验收标准：

- 主要流程可以从新游戏走到赛季结束
- 页面无明显布局错乱
- 关键按钮在非法状态下不可点击
- 无控制台致命错误

## 11. MVP 成功标准

MVP 完成时，玩家应能：

- 打开网页开始一局新赛季
- 从最低级联赛随机获得并管理一支虚构球队
- 查看球员能力与潜力
- 查看球员身价、周薪和合同年限
- 一键生成首发
- 模拟每一轮比赛
- 查看玩家所在联赛与另一级联赛的比分和积分榜
- 每场比赛后看到球员能力变化摘要
- 每场比赛后看到门票收入、工资支出和余额变化
- 进入转会市场购买球员
- 从阵容或转会页卖出球员获得资金
- 推进并完成两级联赛赛季
- 进入新赛季时看到 36 岁球员退役后的阵容变化
- 球队出现空位时自动获得 16 岁年轻球员补充
- 在赛季结束后看到升级、降级结果并开启新赛季

技术上应满足：

- 核心规则集中在可测试的 TypeScript 逻辑模块中
- UI 与游戏规则分离
- 数据结构便于后续迁移到后端数据库
- 本地存档稳定可用

## 13. 建议第一版实施顺序

1. 先完成两级联赛静态数据、模型和页面骨架
2. 再完成玩家球队随机分配、自动首发与比赛模拟
3. 接入球员成长、门票收入、工资支出
4. 接入转会市场、购买球员与卖出球员
5. 然后接入两级积分榜与同轮推进
6. 再加入 localStorage 存档和赛季结束页
7. 最后接入升降级、年龄增长、36 岁退役、16 岁补员、合同年限递减与新赛季生成

原因：先跑通“随机球队 -> 阵容 -> 比赛 -> 财政变化 -> 转会买卖 -> 积分榜 -> 下一轮”的基础闭环，再把赛季末升降级、退役补员、合同年限与新赛季循环接上。
